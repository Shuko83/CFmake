Boost Support

In order to build a project using Boost libraries, following steps are required:

  • set BOOST_ROOT environment variable to Boost root directory path.
  • add CONFIG += boost in your project file to enable Boost support.

Then, header-only Boost libraries will be automatically accessible from your project. Moreover, if you want to use any of the separately-compiled Boost libraries, uses BOOST variable to specify the list of modules which will be linked to the target.

For example:

TARGET = MyApp
TEMPLATE = app

CONFIG += boost   # enables Boost support
BOOST += log      # link to Boost Log library

Project Building

Target Definition

Specifies the name of the target with TARGET variable. For binary targets, optional debug modifiers can be applied to this name by using [...] patterns.

For example, project below will generate MyAppd application in debug mode, and MyApp application in release mode:

TARGET = MyApp[d]
TEMPLATE = app

Following target features will be automatically applied, according to project type:

Project TypeCONFIG FlagsDescription
subdirs-basedno_empty_targetsProject doesn't have targets that do nothing.
orderedVisual C++ projects only: force directories listed to be processed in the order in which they are given. From qmake 3.0 (Qt 5.4.1), this feature has been given up in favour of SUBDIRS .depends modifier use.
binaryx11Linux only: identifies target as a X11 application or library if Qt gui module is used.
skip_target_version_extWindows only: version number is removed on target file name.
applicationwindowsWindows only: identifies target as a windowed application if Qt gui module is used or if target is not identified as a console application.

See also PROPERTIES variable for more details about target properties.

Target Dependencies

Binary target only. Specifies static and shared libraries dependencies by using add_library() function. Prefers using this function to qmake variables, since it automatically manages the related qmake variables assignment according to the type of library. Likewise, specifies plugin libraries dependencies by using add_plugin() function.

According to the current project type, following rules are applied:

  • static library target supports only combination of other archives; any library linked to this target will be ignored.
  • application or shared library target supports archives and shared libraries link; archives combination is not applicable.

Library dependencies support optional debug modifiers syntax, such as described in Target Definition section.

For example:

add_library( shared, $${INSTALLDIR}/shared, MyLib[d] )

Outputs Definition

Binary target only. Below directories are defined to store files generated by the compiler:

  • intermediate generated files root path: generated/$${TOOLKIT.mkspec}/${LIBRARY_MODE}/${BUILD_MODE}.
  • default built output files path: builds/$${TOOLKIT.mkspec}/${LIBRARY_MODE}/${BUILD_MODE}, overridable by using DESTDIR or DLLDESTDIR variable.

Installation Definition

INSTALLDIR variable specifies the root directory for any item path defined in INSTALLS. If not defined, current project file path will be taken into account by default.

For example:

TARGET = MyLib
TEMPLATE = lib
CONFIG += shared

target.path = library    # MyLib deployed in delivery/library

include.files = *.h
include.path = include   # header files deployed in delivery/include

INSTALLDIR = delivery
INSTALLS += target include

Note that QS features implement INSTALLS support for Visual C++ projects, by generating an additional batch file (named project name.install.bat) which provides installation rules. This file implements an install target by build mode if there are more than one (in this case, targets can be passed as batch arguments).
