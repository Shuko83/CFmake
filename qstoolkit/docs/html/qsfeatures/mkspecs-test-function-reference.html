Test Functions

add_subdir(target, path[, depends])

Subdirs-based project only. Specifies a subdirectory target, located in path, that will be registered as part of the projects group. path argument may:

  • be empty. In this case target value will be used as subdirectory path.
  • point to a subdirectory. In this case subproject .pro file should have the same base name as the subdirectory itself.
  • point to a subproject .pro file.

Optional subdirectories targets may be passed in depends argument to define subproject build start constraints. Note that to ensure subdirs dependencies resolution, target should have the same name as subproject's TARGET 'release' value.

add_library(type, path, targets)

Binary target only. Specifies library targets, located in path, to use when linking to / archiving the current target. type should be given to specify the type of dependency:

  • static defines static libraries linked to this target (ignored for static library project).
  • shared defines shared libraries linked to this target (ignored for static library project).
  • combined defines static libraries combined to this target in case of static library project; otherwise is same as static value.

Any library in targets can be defined either by library name (without extension and optional platform dependent prefix), or by file name. Optional [.ext] modifier can also be used in targets values in order to append system-default library extension.

If path is set to system value, this function registers the given system libraries as target dependencies. Note that system path is not supported for combined dependencies on static library project. Furthermore, any pseudo variable ${LIBRARY_MODE} occurrence defined in path argument will be substituted by type argument value.

For example:

add_library( static, $${INSTALLDIR}/static, MyLib )           # links to static library "MyLib"
add_library( shared, $${INSTALLDIR}/shared, MyLib1 MyLib2 )   # links to shared libraries "MyLib1" and "MyLib2"

See also Target Dependencies.

add_plugin(path, targets)

Binary target only. Specifies plugin targets, located in path, dynamically loadable by the current target. Any library in targets can be defined either by library name (without extension and optional platform dependent prefix), or by file name. Optional [.ext] modifier can also be used in targets values in order to append system-default library extension.

Any pseudo variable ${LIBRARY_MODE} occurrence defined in path argument will be substituted by shared value.

For example:

add_plugin( $${INSTALLDIR}/plugins, MyPlugin )   # registers plugin "MyPlugin"

See also Target Dependencies.

assert(condition)

Evaluates given condition. If the condition is false, qmake aborts this project (and its SUBDIRS) processing.

For example, processing of the project below will stop if SOURCES variable is empty:

assert( !isEmpty( SOURCES ) )

build_mode(mode)

Succeeds if the target is being built in the specified mode; otherwise fails. mode can have one of the following values: debug or release.

For example:

build_mode( release ): DESTDIR = "builds/release"
build_mode( debug ): DESTDIR = "builds/debug"

See also build_pass().

build_pass([types])

Succeeds if the target is being built; otherwise fails. If types is specified, this function also checks the type of target currently processed. See TEMPLATE() for more details about types argument.

For example:

TARGET = MyApp
TEMPLATE = app

build_pass(): message( $${TARGET} )        # prints: "MyApp" in debug/release target build pass
build_pass( lib ): message( $${TARGET} )   # never prints: "MyApp"

This function never fails if types is not specified and if debug_and_release / static_and_shared options are disabled.

endsWith(variablename, value)

Tests whether variablename ends with a value that matches the regular expression value.

For example:

endsWith( ICONS, .ico ): \
    message( "$${ICONS} is a valid Windows icon file" )

See also startsWith().

find(variablename, substr)

Tests whether at least one of the values in variablename matches the regular expression substr.

For example:

find( QMAKE_ARGS, ^CONFIG ): \
    message( "qmake arguments defines CONFIG variable" )

glob(variablename, patterns[, base])

Expands the specified wildcard patterns, and adds the resolved list of filenames in variablename. If base is specified, filenames added will be relative to the given base path.

See also rglob().

is_absolute_path(path)

Succeeds if path is an absolute path; otherwise fails if it is relative.

isNull(variablename)

Succeeds if the variable variablename is defined (even if it is empty); otherwise fails.

For example:

isNull( TARGET ): \
    error( "TARGET variable must be defined" )

Note: This function requires qmake 3.0 (Qt 5.0) or later.

is_relative_path(path)

Succeeds if path is a relative path; otherwise fails if it is absolute.

qsdeprecated(string)

Displays string as a deprecation message to the user. Message is prefixed by the processed project name.

qserror(string)

Displays string as an error message to the user and exits. Message is prefixed by the processed project name.

qsmessage(string)

Displays string as a general message to the user. Message is prefixed by the processed project name.

qswarning(string)

Displays string as a warning message to the user. Message is prefixed by the processed project name.

QT(modules)

Succeeds if the QT variable contains the specied modules; otherwise fails. It is possible to specify a regular expression for modules parameter.

For example:

QT( gui ): message( "Windowed application detected" )
else:  message( "Console application detected" )

qtGreaterThan(version[, equals])

Tests that the current Qt version is greater than value version, or equals if equals argument is set to true. Version argument expects a major.minor[.patch] format to be taken into account.

For example:

qtGreaterThan( 5.2, true ): \
    message( "Qt 5.2 or later detected" )

See also qtLessThan().

qtLessThan(version[, equals])

Tests that the current Qt version is smaller than value version, or equals if equals argument is set to true. Works as qtGreaterThan() function.

rglob(variablename, patterns[, base])

Expands the specified wildcard patterns recursively into subdirectories, and adds the resolved list of filenames in variablename. If base is specified, filenames added will be relative to the given base path.

See also glob().

startsWith(variablename, value)

Tests whether variablename starts with a value that matches the regular expression value.

For example:

win32: !startsWith( _PRO_FILE_PWD_, [A-Z]: ) \
    message( "_PRO_FILE_PWD_ defines an absolute path" )

See also endsWith().

TEMPLATE(types)

Tests whether the type of target currently processed matches one of the specified types:

  • aux defines an auxiliary project.
  • subdirs defines a subdirs-based project.
  • app defines an application target.
  • static defines a static library target.
  • shared defines a shared library target.
  • lib defines a static or shared library target.
  • plugin defines a static or shared plugin target.
  • pseudo defines a non-build artifact project (auxiliary or subdirs-based).
  • binary defines a binary file project (application or library).

types argument supports a set of types separated by | character.

For example:

TEMPLATE( static ): INSTALLDIR = "../delivery/static"
TEMPLATE( shared ): INSTALLDIR = "../delivery/shared"

Note that above TEMPLATE values may be queried from a qmake command line, by passing the QUERY_TARGET_PROPERTY=template argument. For example, the following command line returns the type of target defined in the current project:

qmake QUERY_TARGET_PROPERTY=template
